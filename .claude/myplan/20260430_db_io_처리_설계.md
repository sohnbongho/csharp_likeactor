# DB/IO 처리 설계 — SqlWorker + CacheWorker + RedisBroadcast Actor 패턴

날짜: 2026-04-30

## 목표
- MySQL(Dapper) / Redis 작업이 tick 스레드를 블로킹하지 않도록 전용 워커 Actor 구성
- 일반 작업: 채널 만석 시 Drop + 에러 응답
- 중요 작업(로그아웃 등): SqlWorker가 성공할 때까지 재시도
- Redis 이중 사용: 유저 데이터 캐시(cache-aside) + 서버 공지 Pub/Sub 브로드캐스트
- appsettings.json 기반 설정 관리

---

## 전체 흐름

```
[클라이언트]
    ↓ 패킷
[RemoteHandler (tick 스레드)] ── 즉시 반환
    ↓ ISqlRequest / ICacheRequest
[SqlWorkerManager / CacheWorkerManager] ── round-robin 분배
    ↓ Channel<IXxxRequest>
[SqlWorker / CacheWorker (전용 Task)]
    ↓ await Dapper / StackExchange.Redis
    ↓ (실패 시: 중요 요청은 재시도, 일반 요청은 에러 InnerMessage 투입)
[결과 → session.EnqueueMessageAsync(InnerReceiveMessage)]
    ↓
[InnerMessageHandler (다음 tick)] → 클라이언트 응답 Send

──────────────────────────────────────────────────
[서버 내부 / 운영 명령]
    ↓
[RedisBroadcastManager.PublishAsync("notice", message)]
    ↓ Redis Pub/Sub 채널
[RedisBroadcastManager 구독 루프 (백그라운드 Task)]
    ↓
[UserObjectPoolManager.BroadcastAll(noticeMessage)]
    ↓ 모든 활성 세션 Send
[전체 클라이언트]
```

## SqlWorker 설계

### ISqlRequest
```csharp
public interface ISqlRequest
{
    IMessageQueueReceiver? Session { get; }  // null = 세션 응답 불필요 (중요 저장)
    bool IsCritical { get; }                 // true = 성공할 때까지 재시도
    Task ExecuteAsync(MySqlConnection connection);
}
```

### SqlWorker 재시도 로직 (개념)
```
루프:
  request = await channel.Reader.ReadAsync()
  attempt = 0
  while (true):
    try:
      if connection is broken: await Reconnect()
      await request.ExecuteAsync(connection)
      break  // 성공
    catch:
      if !request.IsCritical:
        // 에러 InnerMessage를 세션에 투입 (세션이 살아있으면)
        await session?.EnqueueMessageAsync(new InnerReceiveMessage { Message = new DbErrorMessage() })
        break
      // 중요 요청: 지수 백오프 후 재시도
      delay = min(BaseDelay * 2^attempt, MaxDelay)
      await Task.Delay(delay)
      attempt++
```

### 재시도 대상 예시
| 작업 | IsCritical | 이유 |
|------|-----------|------|
| 로그아웃 유저 정보 저장 | true | 유실 불가 |
| 아이템 획득/소비 저장 | true | 경제적 데이터 |
| 로그인 조회 | false | 실패 시 재접속 유도 |
| 통계 로그 저장 | false | 일부 유실 허용 |

## Redis 이중 사용 설계

### 1) 유저 데이터 캐시 — CacheWorkerManager (Cache-Aside 패턴)

```
로그인 시:
  RemoteHandler
    → CacheWorkerManager.Enqueue(GetUserCacheRequest)
    → CacheWorker: Redis GET "user:{userId}"
      → 캐시 HIT:  세션에 CacheHitResult InnerMessage
      → 캐시 MISS: SqlWorkerManager.Enqueue(LoginSqlRequest)
                    → DB 조회 후 Redis SET "user:{userId}" (TTL 30분)
                    → 세션에 LoginSqlResult InnerMessage

로그아웃 시:
  RemoteHandler
    → SqlWorkerManager.Enqueue(LogoutSqlRequest, IsCritical=true)  // DB 저장
    → CacheWorkerManager.Enqueue(InvalidateCacheRequest)           // 캐시 삭제
```

### ICacheRequest
```csharp
public interface ICacheRequest
{
    IMessageQueueReceiver? Session { get; }
    Task ExecuteAsync(IDatabase redis);
}
```

### 2) 공지 브로드캐스트 — RedisBroadcastManager (Pub/Sub)

**구조:**
- `ConnectionMultiplexer`를 CacheWorkerManager와 공유 (동일 인스턴스)
- Publish: `ISubscriber.PublishAsync(channel, message)` → 직접 호출 (워커 불필요, thread-safe)
- Subscribe: 서버 시작 시 백그라운드 루프 등록 → 수신 시 전체 세션 브로드캐스트

```csharp
// 공지 발송 (서버 내부 또는 운영 도구에서 호출)
await broadcastManager.PublishNoticeAsync("서버 점검 예정: 22:00");

// 내부 수신 흐름 (자동)
Redis → RedisBroadcastManager.OnMessageReceived
      → UserObjectPoolManager.BroadcastAll(
            new MessageWrapper { ServerNotice = new ServerNotice { Message = text } })
```

**UserObjectPoolManager.BroadcastAll 추가:**
```csharp
public void BroadcastAll(MessageWrapper message)
{
    foreach (var kv in _activeSessions)
        kv.Value.Send(message);
}
```

## 주의사항 / 위험 요소

### 중요 요청(IsCritical) 재시도 중 서버 종료
- `TcpServer.Stop()` 시 SqlWorkerManager에 완전 종료 신호를 보내야 함
- 재시도 중인 요청이 있으면 CancellationToken으로 중단 → 로그에 기록
- 필요 시 종료 전 "처리 완료 대기" 타임아웃 설정 권장

### Redis 연결 끊김
- `ConnectionMultiplexer`는 자동 재연결 지원 (`abortConnect=false` 옵션 설정 권장)
- Pub/Sub 구독은 재연결 후 자동 재구독이 되지 않으므로 `ConnectionRestored` 이벤트에서 재등록 필요

### BroadcastAll 성능
- 활성 세션 10,000개에 Send() 동시 호출 시 순간 부하 발생
- `Send()`는 `ConcurrentQueue`에 넣는 것이므로 실제 I/O는 분산됨 — 큰 문제 없을 것으로 예상
- 단, `_activeSessions` 순회 중 세션 추가/삭제는 `ConcurrentDictionary`가 안전하게 처리

### Cache-Aside 동시성
- 두 요청이 동시에 캐시 MISS → 둘 다 DB 조회 → 중복 쓰기 발생 가능 (Cache Stampede)
- 로그인처럼 세션당 1회 발생하는 작업은 문제 없음
- 공유 데이터(월드 정보 등) 캐시 시에는 분산 락(Redis SETNX) 고려

### appsettings.json 민감 정보
- ConnectionString에 패스워드 포함 → `.gitignore`에 추가 또는 환경변수 오버라이드 방식 권장
